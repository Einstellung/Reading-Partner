# 信息源系统：Reading-Partner

> 本文记录"信息源系统"的设计共识，是 [16](./16-知识图谱.md) 的下游。16 定的是分诊、简报交互、画像、反馈；本文只写源——怎么从写死两个源变成用户能订阅任何东西的通用系统。调研全集在 memory 的 info-source-ingestion（五轮实测），本文只落结论。

---

## 定位

产品愿景是 AI 当信息入口。主线是用户用这个软件订阅自己感兴趣的任何东西，内置的源只是出厂预置。适配广泛用户是第一优先级，不是把我们选的几个源做精。

## 源是数据不是代码

一个源是一份声明式配置（source descriptor，JSON），通用引擎执行它。描述进用户数据、随账户同步走。内置源用同一格式表达，不走特例代码路径（dogfooding）。

现状是反面：`InfoSource` 是 `"jiqizhixin" | "qbitai"` 联合类型，两个 `collectXxx` 函数各写死一套端点和解析，`live.ts` 的 `collect()` 把两者拼起来。演进方向：端点和解析策略下沉进描述，`collect(descriptor)` 按管子类型分派；`InfoItem.source` 从联合类型变成源 id 字符串；triage 的 `SOURCE_LABEL` 从描述里取。现有的内部-API 分支和 feed+抓页分支已经是代码，通用引擎是把它们参数化，不是重写。

## 描述的两层

发现层（discovery）和全文层（fulltext）分开声明。发现层回答"怎么拿到条目列表"，全文层回答"正文从哪来"。分开是因为同一发现方式可以配不同全文来源，也因为有些源只有发现层。

管子类型枚举，全部来自实测：

- feed 即全文：feed 自带正文，描述里标正文字段位置（`content:encoded` / `summary` / `description`）。免抓页。
- feed + 抓页：feed 只给标题摘要，正文抓文章页用 Readability 抽。
- 列表页 + 抓页：站点无 feed 但 SSR，解析列表页 HTML 拿文章链接，再抓页抽正文（界面新闻模式）。
- 内部 JSON API：站点背后有公开 JSON 接口，列表和详情端点可配，请求头可配（机器之心、金十模式）。
- 发现层-only：只有标题+摘要，正文拿不到（Bloomberg RSS、HN、arXiv）。分诊按摘要级处理，点开跳系统浏览器读原文。
- 流式快讯：不是文章是快讯流，每条几十字即完整内容，带 `important`、聚类字段（金十模式）。分诊按流处理，不当长文喂。

描述字段还包括：请求身份（UA，私有 header）、付费截断标记（feed 里被截断带 "Read more" CTA 的检测，Substack）、禁抓页标记（计量墙站如 IEEE Spectrum、MIT TR，正文直接用 feed 内容，不去抓文章页）。

后续源类型本期不实现，但格式留位：newsletter-IMAP（邮件全文源）、订阅 webview 按需抽取（已登录 webview 打开文章 URL 抽渲染后 DOM）。

## AI 加源

主线功能。用户在对话里说媒体名或贴 URL，AI 生成描述并入列表。

AI 用探测工具生成描述：试 `/feed`、`/rss`、`wp-json` 等常见路径，判断站点是 SSR 还是 SPA，验证文章页可抓性。这是把五轮调研里 agent 手工干的事固化成工具。生成描述后试运行，真抓三篇，把标题和正文长度给用户看，确认后入列表。探测失败就如实说接不了这个源，不糊弄。

源列表 UI：可见、可开关、可删、可手动贴 RSS URL 直接加。

## 抽取栈

@mozilla/readability 维持主力。加 defuddle（Obsidian Clipper 团队，MIT，纯 JS 吃 DOM，脏页面比 Readability 少裁剪）做兜底，关掉它的 FxTwitter 异步 fallback（`useAsync: false`）保持零外呼。引入前在已接源（量子位、界面、机器之心）的正文页上 A/B，别盲换。

不上小模型：ReaderLM 是非商用 license，且桌面推理慢，不如毫秒级 defuddle。不上 headless，不上云抽取服务。

## triage 配套

两处要改（分诊主逻辑在 16，这里只列源系统带来的增量）：

- 摘要降级：发现层-only 源只有标题+导语也要能分诊。prompt 和 JSON 契约里标注哪些条目是摘要级，让模型知道它读到的不是全文。现有 `formatItem` 已经在正文缺失时回退到 summary，但没告诉模型这是降级；要显式标。
- 跨语言合并：中英文源报同一事件要合并成一条。16 里已有同源去重，这里扩到跨语言同事件。

## 分期

M-info-2：通用引擎 + 描述格式 + 迁移现有两源 + 出厂预置源 + AI 探测加源 + 源列表 UI + defuddle 兜底 + triage 摘要降级和跨语言合并。

M-info-3：

- newsletter-IMAP 源类型：Bloomberg 专栏（Money Stuff / Points of Return 全文邮件）、Substack 付费版、经济学人邮件。IMAP 轮询 + 发件人过滤 + 解析邮件 HTML（Readability 清洗 tracking/CTA）。
- 订阅 webview 按需源：经济学人、Science。已登录 webview 渲染文章 URL，注入 JS 抽 DOM 交给抽取栈。裸 HTTP 过不了它们的 Cloudflare，真浏览器引擎能过。
- Bloomberg 九栏目 RSS 预置：官方 `bloomberg.com/feeds/<栏目>/news.rss` 做发现层，全文走 newsletter。
- 金十降格接入：进每日简报的"今日市场大事"小节，只取 `important=1` 的聚类，不做成独立信息流。

后置（已另记 north star）：播客、视频、推特。放弃：B 站、公众号。

## 红线

- 不绕反爬硬墙。Bloomberg 文章页是 PerimeterX 硬墙，判死，不做自动全文抓取，只用官方 RSS 发现层 + newsletter 全文。
- 订阅内容仅限用户用自己的账号自己读（webview 复用用户已登录会话），不代抓、不共享账号。
- 抓来的全文不公开再分发。个人阅读分诊接近 Reader View 常规用法，红线在分发。
- 私有 API 的鉴权头（机器之心、金十）写成可配置。都是未公开接口，对方有动机收紧，URL 和 header 要能改，机器之心留公众号镜像兜底。
