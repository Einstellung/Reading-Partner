# AI 接入备忘

AI 能力用 pi 提供的 `@earendil-works/pi-ai`。认证范围:Anthropic 走订阅 OAuth(Claude Pro/Max),OpenAI 走订阅 OAuth(ChatGPT Plus/Pro),DeepSeek 走 API key。本文是先导验证(pi-ai 0.80.6,实测)出来的对接手册:怎么打进 webview、请求怎么走 Tauri、OAuth 怎么落地、流式接口长什么样。

## 版本与导入

`@earendil-works/pi-ai@0.80.6`。子路径导出:`.`(主入口)、`./providers/*`、`./api/*`、`./oauth`、`./compat`、`./bedrock-provider`。注意 `anthropicProvider` 不在主入口,在 `@earendil-works/pi-ai/providers/anthropic`;OAuth 相关在 `@earendil-works/pi-ai/oauth`。

## webview 打包

能打进浏览器 bundle。主 `index.js` 没有任何静态 node-only 依赖(aws-sdk/smithy/proxy-agent/`node:` 全部无静态 import),bundle 里出现的 `node:crypto/http/fs/os/stream` 都在 dynamic import 或环境守卫后的懒路径里,不影响加载。实测:headless Chrome 加载无报错,`anthropicProvider()` 实例化成功,`getModels()` 返回模型列表,`streamSimple()` 正常出流。底层是 `@anthropic-ai/sdk`,pi 已给它设 `dangerouslyAllowBrowser`。

## fetch 桥

pi 的所有请求走全局 `fetch`(实测拦截到 `POST https://api.anthropic.com/v1/messages`,带 `x-api-key` 和 `anthropic-beta` 头)。Tauri 里启动时把 `@tauri-apps/plugin-http` 的 `fetch` 顶替全局 fetch,pi 的请求就全走 Rust 侧(绕过 CORS,凭据不进 JS):

```ts
import { fetch as tauriFetch } from "@tauri-apps/plugin-http";
window.fetch = tauriFetch; // WHATWG 兼容签名,drop-in
```

capability 必须手动配 origin 白名单,否则默认拒绝一切。权限集 `http:default`(含 allow-fetch / -send / -read-body / -cancel / -cancel-body):

```json
{
  "identifier": "http:default",
  "allow": [
    { "url": "https://api.anthropic.com/*" },
    { "url": "https://platform.claude.com/*" },
    { "url": "https://claude.ai/*" },
    { "url": "https://api.openai.com/*" },
    { "url": "https://auth.openai.com/*" },
    { "url": "https://chatgpt.com/*" },
    { "url": "https://api.deepseek.com/*" }
  ]
}
```

pi 的 `StreamOptions` 还有 `onPayload` 钩子和响应前回调,需要拦截或改包时用。

## Anthropic 订阅 OAuth

授权码 + PKCE 流程。常量来自 pi-ai `dist/utils/oauth/anthropic.js`:

- client_id: `9d1c250a-e61b-44d9-88ed-5944d1962f5e`(公共 client)
- 授权页: `https://claude.ai/oauth/authorize`
- token 端点: `https://platform.claude.com/v1/oauth/token`
- redirect_uri: `http://localhost:53692/callback`(注册值,精确匹配)
- scope: `org:create_api_key user:profile user:inference user:sessions:claude_code user:mcp_servers user:file_upload`
- PKCE S256,用 Web Crypto(`crypto.subtle` + `getRandomValues`),浏览器可跑;pi 把 PKCE verifier 直接当 `state` 传

授权 URL 构造(`code=true` 让 Anthropic 授权页把 code 显示出来供手动拷贝):

```
https://claude.ai/oauth/authorize?code=true&client_id=<CLIENT_ID>&response_type=code
  &redirect_uri=http%3A%2F%2Flocalhost%3A53692%2Fcallback&scope=<SCOPES>
  &code_challenge=<challenge>&code_challenge_method=S256&state=<verifier>
```

token 交换:`POST` token 端点,JSON body `{ grant_type: "authorization_code", client_id, code, state, redirect_uri, code_verifier }`,返回 `{ access_token, refresh_token, expires_in }`。刷新:`{ grant_type: "refresh_token", client_id, refresh_token }`,可直接用 pi 导出的 `refreshAnthropicToken(refreshToken)`(纯 fetch,浏览器可用)。`getOAuthApiKey(providerId, credentials)` 会自动判断过期并刷新。

token 端点响应不带 CORS 头(实测 bogus code 换 token 返回 HTTP 400 `invalid_grant`,无 `access-control-allow-origin`)。所以 token 交换必须走上面的 Tauri http 桥,webview 直接 fetch 会被 CORS 挡。

### 落地:loopback 自动捕获(定稿,2026-07-13 用户拍板,免手工粘贴)

pi 的 `loginAnthropic` 无条件起 node:http 回调服务器,webview 里必抛,且 pi 没导出 code 交换原语。所以登录前半段自己实现:

1. webview 生成 PKCE + 拼授权 URL(约 15 行)。
2. 通知 Rust 侧起一次性 `localhost:53692/callback` HTTP 监听(约 50-80 行:一次性监听 + 完成后自动关闭 + 超时,拿到 code 后经 Tauri event 发回前端,并给浏览器返回一个"可以关掉此页"的成功页)。
3. `@tauri-apps/plugin-opener` 的 `openUrl(authUrl)` 开系统浏览器,用户登录授权,浏览器自动重定向到 loopback,code 被捕获,全程无手工输入。
4. webview 收到 code,经 Tauri http 桥 POST token 端点换 token。
5. 刷新复用 `refreshAnthropicToken`。

兜底:loopback 端口被占或超时,回退手动粘贴(授权 URL 带 `code=true` 时 Anthropic 页面会显示 code,粘贴进设置页同样能换 token,redirect_uri 照填注册值即可,不需真监听)。

## OpenAI 订阅 OAuth(ChatGPT Plus/Pro)

和 Anthropic 同一姿态:用官方 Codex CLI 的公共 client 走授权码 + PKCE,没有面向第三方的独立正式通道。常量来自 pi-ai `dist/utils/oauth/openai-codex.js`:

- client_id: `app_EMoamEEZ73f0CkXaXp7hrann`
- 授权页: `https://auth.openai.com/oauth/authorize`(额外参数 `id_token_add_organizations=true`、`codex_cli_simplified_flow=true`、`originator=pi`)
- token / refresh 端点: `https://auth.openai.com/oauth/token`(form-urlencoded;token 交换不回传 `state`,与 Anthropic 不同)
- redirect_uri: `http://localhost:1455/auth/callback`(注册值,精确匹配;端口和路径都跟 Anthropic 的 53692/callback 不同,所以 Rust 回调监听器参数化了 port + path,三个 OAuth 调用方共用)
- scope: `openid profile email offline_access`
- access token 是 JWT,内含 `chatgpt_account_id`;pi 的 codex 推理 api 自己从 token 解出来设 `chatgpt-account-id` 头,所以只需把 access token 当 `apiKey` 注入,和 Anthropic 同路。
- 刷新复用 pi 导出的 `refreshOpenAICodexToken`(纯 fetch,浏览器可用);pi 不给 5min skew,我们自己在存储时扣。

provider 用 `openaiCodexProvider()`(不是 `openaiProvider()`),`baseUrl` 是 `https://chatgpt.com/backend-api`,Responses API 形态(`/codex/responses`)。订阅态可用模型限定为 codex 模型集(`gpt-5.4`、`gpt-5.4-mini`、`gpt-5.5`、`gpt-5.6-*` 等),provider 模型列表跟着来,不再是通用 OpenAI 模型。

约束:codex provider 默认走 WebSocket transport,但 webview 的 WebSocket API 带不了 `Authorization` / `chatgpt-account-id` 头,CSP 也禁 wss。所以对 OpenAI 强制 `transport: "sse"`,经同一个 Tauri http 桥出流。落地流程(loopback 自动捕获 + 手动粘贴兜底)与 Anthropic 完全一致,只是端口/路径/token 端点换成上面这套。

存量 OpenAI API key 凭据直接弃用:读到旧格式忽略,不迁移(用户拍板只支持订阅)。

## DeepSeek(API key)

provider 工厂 `deepseekProvider()`,auth 走 `auth.apiKey` 传 key,用 openai-completions 兼容 api,`baseUrl` 指向 DeepSeek 端点。key 存本地(见 CLAUDE.md,绝不入库),请求同样走 fetch 桥。

## 流式接口

```ts
import { anthropicProvider } from "@earendil-works/pi-ai/providers/anthropic";
const p = anthropicProvider();          // 或通用 createProvider({ id, name, auth, models, api })
const models = p.getModels();           // Model[]
const stream = p.streamSimple(model, context, options);  // 或 p.stream(...)
for await (const ev of stream) { /* 增量事件 */ }
const final = await stream.result();    // 最终 AssistantMessage
```

- `context`: `{ systemPrompt?, messages: Message[], tools?: Tool[] }`
- `options`(StreamOptions): `{ apiKey?, temperature?, maxTokens?, signal?, transport?, cacheRetention?, sessionId?, onPayload? }`
- provider 的 `auth`: `{ apiKey?: ApiKeyAuth, oauth?: OAuthAuth }`;OAuth 走 `auth.oauth`,`Models.getAuth()` 内部自动刷新
- 返回的 `AssistantMessageEventStream` 是 `AsyncIterable`,`for await` 消费增量,`result()` 拿终值

事件类型(`AssistantMessageEvent`,每个都带 `partial: AssistantMessage`):

| type | 含义 |
|---|---|
| `start` | 消息开始 |
| `text_start` / `text_delta` / `text_end` | 正文分块 |
| `thinking_start` / `thinking_delta` / `thinking_end` | 思考过程分块 |
| `toolcall_start` / `toolcall_delta` / … | 工具调用分块 |
| `done` | 成功终止,携带最终 AssistantMessage |
| `error` | 失败终止,stopReason 为 error/aborted，带 errorMessage |
