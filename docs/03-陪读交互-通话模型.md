# 陪读交互：通话模型

> 本文档记录"AI 陪读"这件事在界面上到底长什么样的定稿共识，写给"三个月后回来看的自己"。
> 与前三份同一风格：重点不只是最终结论，更包括**被否掉的方案和否掉的理由**。
> 本文是 [01-交互设计.md](./01-交互设计.md) 第二节"四条通道"和第五节"界面布局"的落地展开，并**取代**那两节里的部分早期设想。

---

## 一、核心隐喻：一通电话，不是一个面板

整套交互只用一个隐喻——**微信视频通话**：

- **接通**：用 AI 笔划一句 / 圈一块 → 这一通开始；
- **画中画互换**：书和 AI 谁占主画面，随时点小窗翻转；
- **挂断（✕）**：这一通结束。

这个隐喻替代了最初"点按钮打开 AI 侧栏"的想法。差别不是措辞：

- "打开面板"是**二值的**（开/关），表达不了"AI 一直在陪着我读"这件事；
- "通话中"是一个**持续状态**，它天然携带"有一场对话正在进行"的语义——而这恰恰是"陪读"与"开着一个聊天窗口"的分界线。

由此，整个交互里**没有任何一个"打开某个面板"的动作**，全部是收缩 / 展开 / 互换。

## 二、入口：AI 笔（模式在前，选区在后）

AI 的主入口是**左侧笔架上的一支笔**，跟高亮笔、下划线笔并列。按下它，之后划到哪、聊到哪。

### 被否的方案：选中文字后，在划词浮窗里点"问 AI"

即"选区在前，动作在后"。否决理由有二：

1. **iPad 上这是个反手动作**——浮窗位置随选区飘、按钮小，手指/Pencil 选完还要跨屏去够那个按钮。而"先选笔，再划"完全贴合 Pencil 的肌肉记忆。
2. **工具模式能省掉浮窗那一跳**。按下 AI 笔的那一刻意图已经声明完毕，再弹一个"选颜色 / 高亮 / 下划线"的浮窗纯属碍事。**AI 笔模式下不弹划词浮窗，划完直接进气泡。** 这是工具模式最大的效率来源。

### 这支笔的三个属性（已定稿）

1. **留痕**：AI 笔划出一条视觉上可辨认的专属 annotation（虚线 + ✨ 角标之类，具体视觉待设计）。
   - 为什么必须留痕：**痕迹不是装饰，是索引**。三个月后重翻这本书，看到 ✨ 就知道"这儿我卡住过、跟 AI 聊过"，点一下当年的对话就回来了。
   - 由此确立一个更一般的结论：**划线是一个容器**。一条划线上可以同时挂颜色、批注（打字）、和一场 AI 对话。02 文档里"证据锚点一等公民"在交互层不是一个功能，就是这个——**对话天然长在原文上**。
   - 附带红利：气泡不需要"保存"按钮。它有宿主，关掉也丢不了。
2. **Sticky**：按下就一直是 AI 笔，划五处问五处，直到手动切走。
   - 不做 one-shot（问完自动弹回上一支笔）的理由：Zotero 现有的高亮笔/下划线笔本来就是 sticky 的，AI 笔搞特殊反而破坏肌肉记忆。
   - 误触发的代价很低：误弹一个气泡点掉就完了，不像误画一道墨迹要撤销。
3. **能圈**：这支笔的语义是"**你划什么，我看什么**"，因此同时吃两种选区——划文字（文本上下文）、框选区域（截图，喂多模态，对付 PDF 图表和公式）。
   - 由此 01 文档第二节里分开列的**通道 3（选中问 AI）和通道 4（圈选问 AI）合并为一支笔的两种手势**，不必做两套。原设想的"临时手势层"整个删除——Zotero 已有区域选取工具（image annotation），我们只是把它的出口接到 AI。

### 兜底入口

顶栏保留一个 AI 按钮，用于**无选区提问**（"这一章到底在讲什么"）。它直接进主画面态，不经过气泡。

注意主次：**主入口是选区，不是按钮**。按钮是兜底，不该抢眼。

## 三、三档视图与状态机

①②③ 不是三个功能，是**同一通电话的三档视图**（正如微信的语音小条 / 悬浮小窗 / 全屏通话）。

```
        读书（无通话）
             │  ✨AI笔 划一句 / 圈一块 = 接通
             ▼
   ① 气泡态 ── 贴在划线旁，答案直接开讲（不用打字）
             │  点外面（不是挂断，是继续读）
             ▼
   ② 小窗态 ── 书是主画面，AI 缩成角落小窗   ┐
             │  点小窗                      │ 点任一小窗
             ▼                              │ 即互换
   ③ 主画面态 ─ AI 是主画面，书缩成角落小窗  ┘
             │  点 ✕
             ▼
        挂断 ── 小窗消失，回到纯读书
```

气泡态下 AI **直接开始解释，不要求用户打字**——最高频的意图就是"这句什么意思"，让用户打字是税。答案出来后可在气泡内追问。

### 被否的方案 A：上下分栏

最初设想"点 AI 按钮 → 阅读器上下分栏"。否决理由：**PDF 是固定版式**，竖屏上下切一半后一页只能看半页，读一段话要来回滚——而 PDF 为主是硬需求（00 文档第二节）。EPUB 会重排还能活，PDF 直接废。分栏最多是横屏下的可选态。

### 被否的方案 B：悬浮球

最初设想"退回书时，对话最小化成一个悬浮球"。否决理由：球没有信息量，只是个把手。**画中画互换**才是正确的隐喻——小窗本身就该承载信息，而且互换是对称的、可逆的，符合"书和 AI 谁是主角随时翻转"的真实需求。

## 四、小窗里放位置卡片，不是页面缩略图

```
被否（页面缩略图）          定稿（位置卡片）
┌──────────┐              ┌──────────────┐
│ ▒▒▒▒▒▒▒▒ │              │ 📖 眼与脑     │
│ ▒▒▒▒▒▒▒▒ │  ← 看不清     │ 第5章 · p87   │
│ ▒▒▒▒▒▒▒▒ │              │ "…神经节细胞  │
│ ▒▒▒▒▒▒▒▒ │              │  越密越清晰"   │ ← 最后划的那句
└──────────┘              └──────────────┘
```

否决缩略图的理由：微信那个小窗放的是你自己的脸，是有信息量的；但**书页缩略到那么小，字根本读不出来**，信息量趋近于零，只剩一个"回到书"的把手。

位置卡片在聊天时是真的有用：聊到一半忘了原文说啥，瞄一眼就有；点一下回书里。

## 五、通话边界 = memory 蒸馏的物理开关（本轮最大收获）

### 悬案：同一本书里在 20 处划线问 AI，是 20 场对话还是 1 场？

- 若是 **20 场孤立问答**：AI 在第 10 次提问时不知道你前 9 次卡在哪——那它不是陪读，只是划词翻译。
- 若是 **1 场无限长的流**：几百轮之后无从翻阅，且没有任何自然边界。

### 定稿：若干通电话。通内连续，通间靠 memory 连续。

这个答案是**从"挂断"这个交互本身推出来的**——✕ 的语义是"结束**本次**对话"，隐喻里天然自带边界。

跟现实完全一致：你跟一个朋友打过 20 通电话，每通有始有终、各有主题，但你们的关系是连续的，第 20 通时他记得你三个月前说过什么。**"陪读"要的从来不是一条无限长的聊天记录，是这个。**

于是两层连续性各归其位：

- **通内**：同一条 transcript，AI 记得你这通里前面问过什么；
- **通间**：靠 memory，AI 记得你三周前的立场（01 第三节第二层）。

### 由此解掉的一个老问题

02 文档定了"会话收尾静默写入：会话结束时触发静默 agent 回合，把值得留的写进 memory"——但**"会话结束"在 UI 上一直是个虚的东西**：关 App 算吗？切书算吗？发呆十分钟算吗？只能靠猜。

现在不用猜了：**挂断 = 这一通结束 = AI 后台静默消化这一通，蒸馏进主题 memory。** 01 文档里"合上书，AI 后台消化"的设想，在这里找到了它的物理开关。

### ✕ 不需要二次确认

挂断没有破坏性——transcript 和 memory 都留着，重新划一句就是新的一通。弹确认框只会打断心流。

## 六、通话跨材料存活

**一通电话属于"主题"，不属于"书"。**

在 A 书里接通，聊着聊着去翻 B 论文对照——**这一通继续**，小窗里的书自动换成 B。

理由：01 第一节定的顶层容器是主题不是书。而且这恰恰是主题阅读最爽的一刻：AI 能说"你现在翻的这篇，做法和你刚才那本第三章是矛盾的"。**这才是 AI 伴读该有的样子。**

## 七、跨端：语义层一致，表现层不同

- **语义层（跨端一致）**：接通 / 通话中 / 挂断 / 通话边界触发 memory 蒸馏。
- **表现层（端相关）**：
  - **iPad 竖屏**：画中画互换（本文第三节）；
  - **桌面 / 横屏**：屏幕够大，不需要互换——"通话中"表现为 AI 栏常驻展开，即 01 第五节的三栏布局。

同一通电话，两种显示方式。

## 八、待定事项

1. **长通话的 memory 蒸馏时机**。通话既然能跨书存活，一通可能长达数小时、横跨多份材料，光靠"挂断"触发蒸馏可能太晚。补充触发点：**上下文将压缩时也蒸馏一次**（02 文档从 OpenClaw 抄来的分层蒸馏本就有这条）。挂断 = 主信号，上下文压缩 = 兜底信号。
2. **AI 笔痕迹的具体视觉**（虚线 + ✨ 只是方向，未做视觉设计）。
3. **气泡里是否提供快捷意图**（解释 / 举例 / 与前文的关系）——偏功能，暂缓。
4. **AI 始终知道的上下文，要不要对用户可见可编辑**（一条常驻的"📎 上下文"条，显示当前章节页码、刚划的句子、召回的 memory 条数，可一键增删）。讨论中提出但**主动押后**——属于功能层，等交互流程定完再谈。

## 九、实现层备注（不展开，只记账）

- 你在 iPad Zotero 里看到的那套 UI（左侧竖排笔架、顶栏）**一行都不属于 reader 引擎**。`src/index.ios.js` 的入口是 `createView` 而非 `createReader`，只挂载一块"渲染 + 批注"的画布，工具栏/侧栏/划词浮窗全是 Zotero 用 Swift 画的原生 UI（划词浮窗甚至是 `onSetSelectionPopup` 把坐标 postMessage 给原生去弹）。
- **我们拿的是引擎，不是那个界面。壳的 UI 100% 自己画。**
- 好消息：触控与 Pencil 手势（`pointerdown` / `touchstart`）都在 **web 层**（`pdf-view.js`、`dom-view.tsx`、`epub/flow.ts`、`annotation-overlay.tsx`），不在原生层。用 web 构建打进 Tauri，iPad 上照样能划能圈。
- 因此"AI 笔"这支笔要自己画工具栏、自己管模式——这可能把我们推向 **`createView` 而非 `createReader`**，即走 Zotero iPad 版的同一条架构路子（引擎只做画布，chrome 全自建）。**待验证。**
