# 陪读交互：通话模型

本文是 [01-交互设计.md](./01-交互设计.md) 第二节"四条通道"和第五节"界面布局"的落地展开，并取代那两节的部分内容。

## 隐喻

整套交互按微信视频通话来组织：用 AI 笔划一句或圈一块＝接通；书和 AI 谁占主画面，点小窗互换；点 ✕ 挂断，本次对话结束。

没有"打开 AI 面板"这个动作，全部是收缩、展开、互换。

## 入口：AI 笔

主入口是左侧笔架上的一支笔，跟高亮笔、下划线笔并列。按下之后划到哪聊到哪，模式在前、选区在后。AI 笔模式下不弹划词浮窗，划完直接进气泡。

三个属性：

1. 留痕。AI 笔划出一条可辨认的专属 annotation（虚线加角标，具体视觉待设计）。痕迹是索引：重翻这本书时点它，当年的对话就回来了。由此确立一个更一般的结论——划线是一个容器，一条划线上可以同时挂颜色、批注和一场 AI 对话。气泡因此不需要"保存"按钮。
2. Sticky。按下就一直是 AI 笔，直到手动切走。跟高亮笔行为一致。
3. 能圈。这支笔同时吃两种选区：划文字（文本上下文）、框选区域（截图，喂多模态，对付 PDF 图表和公式）。01 文档里的通道 3 和通道 4 合并为这一支笔的两种手势。区域选取直接复用 Zotero 已有的 image annotation 工具。

顶栏另有一个 AI 按钮，用于无选区提问（"这一章在讲什么"），直接进主画面态，不经过气泡。它是兜底，主入口是选区。

## 三档视图

①②③ 是同一通电话的三档视图，不是三个功能。

```
        读书（无通话）
             │  AI 笔划一句 / 圈一块 = 接通
             ▼
   ① 气泡态 ── 贴在划线旁，答案直接开讲
             │  点外面（不是挂断，是继续读）
             ▼
   ② 小窗态 ── 书是主画面，AI 缩成角落小窗   ┐
             │  点小窗                      │ 点任一小窗
             ▼                              │ 即互换
   ③ 主画面态 ─ AI 是主画面，书缩成角落小窗  ┘
             │  点 ✕
             ▼
        挂断 ── 小窗消失，回到纯读书
```

气泡态下 AI 直接开始解释，不要求用户打字。答案出来后可在气泡内追问。

## 小窗内容：位置卡片

小窗放的不是页面缩略图（缩到那么小字读不出来），而是当前位置卡片：书名、章节页码、最后划的那句话。聊天时忘了原文说什么，瞄一眼就有；点一下回书里。

```
┌──────────────┐
│ 眼与脑         │
│ 第5章 · p87   │
│ "…神经节细胞  │
│  越密越清晰"   │
└──────────────┘
```

## 线程模型：对话的单位是划线

一条 AI 笔划线＝一个独立聊天线程，像 ChatGPT 的会话列表。点星标，线程回来——那句原文、当时的问答都在，可以接着聊。不是历史记录，是能继续的对话。

上下文分两层：

- 共用层：主题/书级别的上下文（memory），所有线程都躺在它上面。第 10 次提问时 AI 知道前 9 次卡在哪，靠的是这层，不是把所有对话挤进一条 transcript。
- 线程局部：那条划线的原文、周边上下文、这个线程自己的问答历史。

由此几件事的语义就定了：

- 挂断（✕）挂的是通话视图，线程本身留在划线上，随时点星标重新接通。不需要二次确认。
- sticky 笔划第二处＝开第二个线程，上一个自动收起成星标。不存在一通电话里多个话题搅在一起。
- 线程挂断时触发 memory 蒸馏。02 文档定的"会话结束时静默写入"，"会话结束"就是挂断。

## 线程跨材料存活

线程锚在发起它的那条划线上，但锚点只管"从哪来"，不管"能聊多远"：线程活着的时候可以翻到主题内任何材料接着聊，翻页不切线程，小窗里的位置卡片跟着换。只有用 AI 笔划新的一处才开新线程。

AI 因此能说出"你现在翻的这篇，做法和刚才那本第三章矛盾"——这靠共用层，属于主题，不属于书。

## 跨端

语义层跨端一致：接通、通话中、挂断、边界触发蒸馏。

表现层按屏幕大小分：iPad 竖屏用画中画互换；桌面和横屏屏幕够大，"通话中"直接表现为 AI 栏常驻展开，即 01 第五节的三栏布局。

## 待定

1. 蒸馏时机。线程比早先设想的"长通话"短得多，挂断触发基本够用；上下文将压缩时再蒸馏一次作兜底（02 文档从 OpenClaw 抄的分层蒸馏本就有这条）。
2. 共用层里具体放什么、怎么注入（memory 蒸馏与召回的细节），属于 AI 部分的设计，另聊。
3. AI 笔痕迹的具体视觉。
4. 气泡里是否提供快捷意图（解释 / 举例 / 与前文的关系）。
5. AI 的上下文是否对用户可见可编辑（一条常驻的上下文条，显示当前章节页码、刚划的句子、召回的 memory 条数，可增删）。押后到功能层再谈。

## 实现备注（待验证）

iPad Zotero 里那套 UI 不属于 reader 引擎。`src/index.ios.js` 的入口是 `createView` 而非 `createReader`，只挂载一块渲染加批注的画布；工具栏、侧栏、划词浮窗全是 Zotero 用 Swift 画的原生 UI（划词浮窗是 `onSetSelectionPopup` 把坐标 postMessage 给原生去弹）。我们拿的是引擎，壳的 UI 自己画。

触控与 Pencil 手势（`pointerdown`、`touchstart`）在 web 层（`pdf-view.js`、`dom-view.tsx`、`epub/flow.ts`、`annotation-overlay.tsx`），不在原生层。用 web 构建打进 Tauri，iPad 上能划能圈。

AI 笔需要自己画工具栏、自己管模式，这可能把我们推向 `createView` 而非 `createReader`，即走 Zotero iPad 版的同一条架构路子（引擎只做画布，chrome 全自建）。待验证。
