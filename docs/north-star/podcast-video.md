# 播客与视频

## 愿景

独立的一条线，是 info 线和读书线的交集。入口像 info：外部内容流进来，先要分诊值不值得花时间。体验像读书：长内容、共读、批注。核心动作是蒸馏——两小时的访谈转写后，AI 告诉你值不值得看、讲了什么、哪几分钟值得跳看，并能就内容对话。

只做长访谈、演讲、播客这类没有文字替代、信息密度高的内容。新闻类视频不做：文字媒体已经全覆盖，不值得为它维护脆弱的依赖。

## 为什么现在不做

后置，等 info 源系统立起来再排。

分两步。第一步手动喂链接：用户扔一个 YouTube 或播客 URL，AI 拉字幕或转写音频，再蒸馏加共读，复用已有的 URL 摄入交互，先验证真需求。第二步才做订阅式：播客 RSS 先行，YouTube 频道跟上，进每日简报。

## 将来做时已知的事实（2026-07 调研）

- YouTube 字幕唯一稳的管子是捆一个 yt-dlp sidecar（公有领域 license，20–40MB）。它会周期性小坏，所以只在用户主动触发的场景用，不进后台订阅。
- 播客 RSS 原生给音频，转写走已有 STT 管线，是最轻的管子。
- B 站要养号，放弃。
- 成本结构上这条线烧的是转写额度和长文本 token，不是抓取。
