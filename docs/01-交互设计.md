# 交互设计：Reading-Partner

> 本文档记录产品交互设计的定稿共识，写给"三个月后回来看的自己"。
> 与 [00-项目共识.md](./00-项目共识.md) 同一风格：重点不只是最终结论，更包括**被否掉的方案和否掉的理由**。核心机制"三层记忆模型"经历了三版演进，第三节完整记录了这个过程。

---

## 一、信息架构的第一性设计：顶层容器是"主题"，不是"书"

用户的真实场景是**主题阅读（syntopical reading）**：带着一个问题读一批材料——几篇论文 + 几本书。

所以建的不是书架，是**"主题项目"**，里面挂多份 PDF/EPUB。由此推出几条结构性结论：

- **AI 的记忆和笔记都长在主题上，不是书上**——跨材料连续。AI 能说出"这篇的做法和你上周读的那本第三章矛盾"这种话。
- 最终产出的笔记和 PPT 是**主题级的一份**，不是每本书一份。
- 这是与微信读书 AI、NotebookLM 等工具**最本质的差异点**。

## 二、陪读闭环

### 读之前（开题）

新建主题时用户说明：为什么读、想回答什么问题。AI 据此生成阅读问题清单。

注意：**这一步必须足够轻（一句话也行）**，否则会变成没人用的仪式。

### 读的过程中：内容进入 AI 的四条通道

按动手成本从低到高：

1. **划线**——用户的自然动作，AI 自动可见；
2. **划线 + 批注（打字）**——AI 可针对用户的批注回应，不只是原文；
3. **选中 →"问 AI"（划词菜单入口）**——即问即答；
4. **圈选问 AI（Pencil 圈一块区域）**——对付 PDF 图表和公式。技术上是临时手势层，不存墨迹。

关键设计：**AI 始终知道上下文**——当前页码、刚划的内容、主题、memory。用户问一句"这里没看懂"就够，零复制粘贴。这是"陪读"与"开着聊天窗口"的根本区别。

### AI 行为原则

AI **默认不主动插话**——读的心流神圣，只在被划线/提问时响应。

"主动模式"（如读完一节主动提问检验理解）可做成开关，**默认关**。

## 三、核心机制：三层记忆模型（最重要的设计决策，经历三版演进）

### 被否的第一版：逐条审批制

AI 每条回答下带"存入笔记"按钮，人逐条筛选进笔记。

**否决理由**：把沉淀成本摊到每次对话上——读得越投入打断越多，破坏陪读心流。

### 被否的第二版：自动合并制

AI 定期把对话自动"改写合并"进人类可见的主题笔记，人用 diff 提示 + 直接编辑保持掌控。

**否决理由**：AI 直接写人的笔记，需要 diff 提示、编辑权限规则等一堆补丁机制；且笔记的最终形态不应由 AI 单方生成。

### 定稿的第三版：三层模型

映射 Claude Code 的 transcript / MEMORY / 交付文档三件套。

**第一层：对话流**——原始记录，完整保留，不加工。

**第二层：AI 自己的笔记（memory）**——给 AI 自己用的，不是人类交付物。AI 持续维护：读到哪、用户卡在哪些概念、⭐过什么、每份材料核心立场、用户与材料的分歧点。

- **存在目的：跨会话的陪读连续性**。主题横跨几周/几本书/几十场对话，原始对话流放不进上下文；有 memory，三周后翻开第三本书，AI 仍记得"你读第一篇论文时对这个假设持怀疑态度"。
- 人可随时点开查看（建立信任、排查理解偏差），但按 AI 自己好用的方式组织，不追求文笔。
- **更新时机完全自动化**：会话结束后台静默更新，无需确认、无需 diff 提示——因为不是人的交付物，第二版里的那些补丁机制整个不需要了。
- memory 记错了怎么办：人不直接编辑，在对话里纠正（"你记错了"），AI 自己修正。**memory 归 AI 所有**。

**第三层：最终笔记（人触发，人机共创）**——用户想整理时才整理（读完一本书、主题读完、或某天有感觉了）。触发后进入共创会话：AI 基于 memory + 划线批注 + ⭐对话起草稿骨架，然后人机在编辑器里一起迭代（"这段展开""这两点合并""这个观点不对，我的理解是……"）直到满意。笔记每条带**出处锚点**（哪本书哪页哪条划线，可跳回原文）。PPT 从这份笔记生成。

### 该结构的红利

- 最终笔记诞生于用户全程在场的共创，每个字都过目——**"笔记变成 AI 话语堆积场"的担忧被结构性解决**；
- memory 不面向人，可以自由混乱。

AI 核心选型（pi）与第二层 memory 的设计定稿，见 [02-AI核心与memory设计.md](./02-AI核心与memory设计.md)。

## 四、人的信号：⭐重点关注

- 用户读书时的全部动作：划线、批注、提问、偶尔点⭐——**全是自然动作，零管理负担**。
- ⭐的语义："这段值得记住"。同时喂两头：memory 中权重最高；共创起草稿时优先采用。
- 直接"存笔记"的直通道保留，但预期低频。
- 划线和批注天然是高权重信号（用户亲手留下的思考痕迹），沉淀权重高于 AI 自己的话。

## 五、界面布局

桌面端三栏：

```
┌─────────────┬────────────┬────────────┐
│ 阅读区       │ AI 陪读     │ 主题笔记    │
│ (reader引擎) │            │            │
│ 划线/批注    │ 上下文对话  │ 结构化文档  │
│ 圈选手势     │            │ 每条带锚点  │
└─────────────┴────────────┴────────────┘
```

iPad 上：笔记栏收起，阅读 + AI 两栏为主。

**共创会话是独立工作模式**（非聊天内进行）：左边素材面板（memory、⭐对话、划线批注，可点引），右边成形中的笔记文档，中间对话驱动修改。

## 六、完整闭环一句话

读（划线/批注/提问/⭐）→ AI 默默维护 memory → 人想整理时发起共创 → 最终笔记 → PPT。

人的管理负担全程为零，掌控点只在共创那一刻——**那正是人本来就想亲自在场的时刻**。
