# 信息陪读与分诊：Reading-Partner

> 本文记录"信息陪读与分诊"的需求共识，与 [00](./00-项目共识.md)、[02](./02-AI核心与memory设计.md)、[14](./14-笔记与PPT.md) 同一风格。

---

## 定位

入口反转。现在是用户去信息流里找，改成 AI 替用户当信息入口，用户只做"点头"确认。核心不是收集是分诊：AI 先替用户读一遍，分三档——忽略、讲两句就够、值得精读。

灵魂是记忆。筛选依据是这个人（兴趣、读过什么、还没解决的问题），靠 memory，不靠图谱。

## 三层结构

分诊管线的形状，记忆驱动。

- 按天信息流：AI 每天收，每条都读过、写一句摘要、按日期留档不删，看不看都在。
- 提名：AI 从当天信息挑几条建议进某主题并给理由。AI 只能提名不能自塞。提名不过期，攒着；主题共读时的"翻旧账提醒"是提名的第二次机会。
- 主题：成员资格只由用户确认。主题内条目分两档——精读件（进书架走完整共读）和收录件（只留 AI 摘要当背景事实，随时可升级成精读）。

## 记忆驱动的分诊 + 轻量学习路径

分诊：AI 拿 memory（兴趣、读过什么、未解决的问题）判断进来的东西该忽略／略讲／精读，不依赖任何图谱结构。用户要的是别被淹。

轻量学习路径（小白场景）：走一步看一步的下一步建议，跟踪读过什么、下一步读什么，配一张粗略总览兜底方向感。不是锁死的课程表，不是可缩放图谱。价值在指出边界（还没碰的），不在重画已懂的。

## 探索与结论：可视化图谱是伪需求

探索过把知识图谱做成用户盯着看、缩放浏览的产品界面（城市地图式的连续语义缩放、可导航的关系图、按主题存的图数据模型），并抽了一版 14 节点的粗粒度概念图验证抽取。判定砍掉，理由：

- 学习用途：用户真理解了，那张结构图就在他脑子里，PPT 能拆开讲就是随时能外化的证明。把已经懂的画成图给他看是冗余。外部表示唯一多给的是边界（还没学的、不知道自己不知道的），价值在显示空白，不在显示填满处，而且用完就扔。
- info 用途：分诊"这条值不值得读"靠记忆（兴趣 + 读过什么 + 未解决的问题），不靠图谱。把每条 info 分门别类塞进一张图是很重的机器，回报不清楚。

"用户知识边界的结构模型"若将来证明有用，退成 AI 内部、看不见的轻量覆盖度账本，最多小剂量冒头（偶尔提一句"某主题里 X 你还没碰过"），不做成可导航的大件。

## 简报

门厅：app 首页是轻量"今天"页，两张卡并排——"继续读"（上次的书，一点回到原位）和"今日简报"。谁是主角由使用投票，不由设计预设；简报若真成刚需会自然长成入口。不把 app 重心压在未经日常使用验证的功能上（知识图谱的教训）。

简报是一页有底的文档，不是无底的流。四段：

- 一句话总评，允许诚实地说"今天没什么值得看"。
- 值得读，2-4 张卡：标题＋给你的理由＋来源。理由默认露出，既是个性化的证据，也是纠错的靶子。点卡进文章阅读页，读完带已读态。
- 一句话档：那行字就是全部消费，不需点开。
- 滤掉区默认折叠：类别×数量一行，展开见标题列表，每条一键翻案（打开＋告诉画像杀错了）。

另留 0-1 条"重要但不在你日常关注"的破圈位，防回音壁。

反馈只有两种手势：打开（最强正信号，不设点赞）和摁掉（×，这类别少给我）。触屏左滑，桌面 hover。

聊天不做页面底部输入框，用悬浮可缩放聊天窗（微信通话式：收起是小球，展开是完整面板），按条锚定——点某条的"问一下"，聊天带该条上下文打开，和阅读器"划线起 thread"同一模型，一条消息一条 thread。简报是 AI 读完全文后写的，可以就地追问（"Kimi 那条展开讲讲"），不用点开原文。

节奏：不推送、不弹窗、不红点催读。新闻过期就让它过期。

画像不让用户填表定义"我在乎什么"（用户自己说不清，经典参与度推荐系统与产品立意冲突）。画像从用户对简报的反应里学：打开／摁掉／翻案是最肥的信号，聊天里随口说的偏好次之，阅读积淀做种子。目标函数是"值不值得占用注意力"，不是参与度。

## v1 边界

- 只有今天的简报；历史、搜索、AI 回捞、日期列表后置。
- 文章页无划线批注（HTML 锚定是另一套系统，后置）；不进笔记／slides。
- 源写死两个：机器之心、量子位。无源管理 UI。新智元验证过可接（官网 aiera.com.cn 的 WordPress REST API `wp-json/wp/v2/posts` 给全文，feed 挂了），后面看情况再加。
- 不加新设置项，不做推送。

## 和现有系统的接口

精读走现有 reader + notes + 共读，输出走现有 slides 管线。分诊和学习路径都建在 memory 上。上游收集（RSS／arxiv 等）后置。

## 落地：M-info-1

源接入，全部实测过：

- 机器之心走站点内部 JSON API：列表 `api/article_library/articles.json?sort=time&page=1&per=N`，详情 `api/article_library/articles/{slug}.json` 给全文 HTML；须带浏览器 UA。未公开接口，URL／UA 做成可配，留意对方收紧。
- 量子位走原生 WordPress feed（`qbitai.com/feed`，须带 UA，只给标题），全文＝抓文章页＋本地 @mozilla/readability（webview 原生 DOMParser）抽正文。
- 通用结论：不押注通用转换器（Folo／RSS3 底层是 RSSHub，覆盖不了这些源），每个重点源写轻量适配器，优先反向内部 API；"SPA 必须 headless"是伪命题。

管线 `src/info/`：适配器（纯解析、注入式可测）→ 全文补齐 → 净化（文章 HTML 去脚本，图片 no-referrer）→ triage 一次 AI 调用（喂画像＋反馈尾巴＋各条全文截断，产出严格 JSON：总评／值得读＋理由／一句话档／破圈／滤掉＋类别，跨源同一事合并），看门狗共用 `src/ai/watchdog`。

存储：`briefing-<日期>.json` 本地缓存，不同步；画像 `info-profile.md` 和反馈日志 `info-feedback.jsonl` 进同步范围。画像 v1 静态种子＋反馈尾巴直接进 prompt，自动蒸馏后置。

法律姿态：抓公开新闻站做个人阅读分诊，接近 RSS 阅读器／Reader View 的常规用法；红线是不把抓来的全文公开再分发。
