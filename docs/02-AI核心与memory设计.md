# AI 核心与 memory 设计：Reading-Partner

> 本文档记录 AI 核心（agent 框架）选型与 memory 机制的定稿共识，写给"三个月后回来看的自己"。
> 与 [00-项目共识.md](./00-项目共识.md)、[01-交互设计.md](./01-交互设计.md) 同一风格：重点不只是最终结论，更包括**被否掉的方案和否掉的理由**。本文对应 01 中三层记忆模型的第一、二层（对话流 + AI 自己的 memory）的技术落地。

# 第一部分：AI 核心选型——pi（badlogic/pi-mono）

## 选什么

采用 pi 的底层两个库：

- **`pi-ai`**：统一 LLM 接口——30+ provider、流式输出、工具调用、TypeBox schema 校验；
- **`pi-agent-core`**：agent 循环。

**不用 `pi-coding-agent`**——那是一个 Node 终端产品，不是我们要的库。

## 采纳理由

1. **pi-ai 明确支持浏览器环境**（核心代码无副作用，可打进前端 bundle）。我们零后端，AI 循环必须跑在 Tauri webview 里，iPad 无 Node 是死约束——pi-ai 正好匹配这个约束。
2. **MIT 协议**，与 AGPL 的壳（zotero/reader）兼容。
3. **模型可插拔**（Claude/GPT/Gemini/DeepSeek/Kimi/Ollama）：陪读可用便宜模型，共创用最强模型。
4. **我们的 AI 是 agentic 形态**——回答前可能要调工具取上下文、检索材料。自写"流式 + 工具调用 + 重试"循环是几百行易错胶水，pi 把这层做掉了。

## 分工架构

```
你的 AI 层
├─ 陪读 persona 和 prompt（开题、解释风格、共创模式）   ← 自己写
├─ 上下文装配：每轮自动注入页码/选区/最近划线/memory   ← 自己写，产品灵魂
├─ 工具集：读取阅读上下文、检索主题材料、查批注、
│         读写 memory、生成出处锚点、共创编辑笔记      ← 自己写，产品灵魂
└─ pi-agent-core + pi-ai（agent 循环、模型接入）        ← 白拿
```

与 zotero/reader 完全对称：reader 是阅读引擎，pi 是 agent 引擎，都是白拿的核，**产品长在连接层上**。

## 已知注意点（如实记录）

1. **API key 放客户端**：BYOK（Bring Your Own Key），本地应用模式，同 Obsidian 插件的做法。将来若公开发行再加代理后端。
2. **CORS**：webview 直连各家 API 可能受限。解法是 HTTP 请求走 Tauri Rust 侧发出——这可能是第一个要碰原生层的地方，但量很小。
3. **pi 项目年轻、API 会变**：锁版本应对；MIT 协议下最坏情况 vendor 进来自维护。
4. **pi-agent-core 的浏览器兼容性需在 demo 阶段实测**（pi-ai 已确认支持浏览器，agent-core 未确认）。兜底方案：基于 pi-ai 自写精简 agent 循环。

## pi 自带什么、不带什么

- **自带**：会话持久化（append-only JSONL 树，支持恢复/fork/分支）和上下文压缩（compact）——可作为三层记忆模型第一层"对话流"的实现参考。
- **不带**：跨会话长期 memory。但这不是缺陷——memory 本就该是产品层设计而非框架功能（Claude Code 的 memory 也只是"文件目录 + prompt 规则 + 读写工具"，没有什么深层基础设施）。

# 第二部分：memory 设计

> 落地状态（2026-07-14）：memory 整体后移至北极星（[north-star/memory.md](./north-star/memory.md)），本部分是届时的设计依据。先行的 M6 只做"让 AI 看得见书"：全文提取缓存（按页，带 PDF 大纲）、开场上下文装配（书名/章节/选区周边原文/主题书单，实时算不存储）、agent 工具（翻页读原文、主题内全文 BM25 检索、读划线批注）、工具调用过程在聊天中可见；prompt 约束为用户提到才查、不主动发散。历史对话不进 M6 检索。agent 工具循环（本文第一部分的 agentic 形态）随 M6 落地。

## 决策背景：否掉重型 memory 引擎路线

曾调研过 Hindsight（vectorize-io）、LangMem 等开源 memory 引擎（该调研文档已删除，此处记录结论以免重新踩坑）：

- **Hindsight 机制与需求匹配度确实高**——bank 映射主题、observation 自动合并、证据锚点、矛盾修订，几乎条条对上。但它是**自托管服务端**（Docker/K8s + Postgres/pgvector + 后台 worker，无嵌入式/浏览器模式）。采纳它等于推翻三条既定架构决策：零后端、纯 TS 单语言（LangMem 更是 Python）、iPad 平权。
- **规模错配**：那套机制（事实抽取、向量召回、时态矛盾追踪）是为平台级多用户海量事件设计的；我们是单用户、单主题作用域，一个主题几十场会话/几百条划线，Claude Code 式的文件方案完全够。
- **决议**：第一版自建轻量 memory 引擎；重型引擎降为二阶段候选——将来若因多端同步等原因后端反正要出现，再做对照 PoC，用数据证明值得再换。

### 从该调研保留下来的引擎无关原则（照单全收）

1. **边界纪律**：原始记录是事实源（业务库，完整不可丢）、memory 是可随时重建的派生视图、最终笔记是人的作品——memory 引擎坏掉不应损坏另外两层。
2. **Memory Adapter 窄接口**（retain/recall/listObservations/correct/rebuild）：业务层不感知引擎，使引擎可替换。
3. **两条写入通道**：高信号事件即时写——划线、批注、⭐，以及用户在对话里明确表达的观点；会话 transcript 在挂断时批量蒸馏，不逐条触发。
4. **两段式读取**：进主题时注入短快照；提问/共创时按需召回。
5. **四段式上下文装配**：当前阅读上下文（实时取，不进 memory）/ 主题短快照 / 按问题召回的 memory（带来源 ID）/ 原始证据（按需回查）。
6. **不采用的捷径**：
   - 纯向量 RAG——不会维护"用户以前信 A 后来纠正为 B"的演变，也分不清用户观点/材料观点/AI 回答；
   - 全量 transcript 常驻 prompt——成本 + 注意力稀释；
   - 让 memory 直接当最终笔记——重新引入已否决的"AI 单方写人的笔记"（见 01 第三节被否的第二版）。

## 定稿方案：结构学 Claude Code，管线学 OpenClaw，领域层自己做

参考了两个主流 agent 的 memory 机制。两者均属"文件型、agent 自维护、prompt 驱动"流派，且都不是可安装的库——**抄的是设计，不是代码**。

### 从 Claude Code 抄结构

- 按主题隔离的记忆目录（对应它的按项目隔离）；
- 索引 + 明细两段式：索引常驻（限制行数），明细文件按需读；
- 每条记忆带类型/元数据、一事一文件、去重纪律（有相关就更新不新增、错了就删）——memory 是"当前理解的最新版"，不是日志堆积。

### 从 OpenClaw 抄管线

- **分层蒸馏**：transcript 存档 → 会话笔记 → 定期蒸馏进长期 memory、清过期条目；
- **会话收尾静默写入**：触发静默 agent 回合，把值得留的写进 memory——即 01 中"合上书，AI 后台消化"的实现机制。触发点是挂断，即用户点 ✕ 结束本次对话（见 [03-陪读交互-通话模型.md](./03-陪读交互-通话模型.md)）。补充触发点是上下文将压缩时：一通对话跨材料存活，可能长达数小时横跨多本书，只靠挂断触发太晚。挂断是主信号，上下文压缩是兜底；
- **时间衰减**：带日期记忆衰减、常青文件不衰减。映射到本项目："读到第几页"类进度信息该衰减，"用户对某假设持怀疑"类理解立场该常青；
- **混合检索**（memory 长大后）：切块 embedding + BM25 + MMR；社区已抽出独立库 memsearch 可评估。第一版数据量小，可只用 BM25 关键词检索甚至纯快照。

### 两家都没有、属于我们领域层的

- **证据锚点一等公民**：每条 memory 指回划线/批注/消息 ID，可跳回原文。锚点由交互天然给出，不需要额外机制建立——AI 笔划出的划线就是对话的宿主，一条划线上同时挂着颜色、批注和一场 AI 对话（见 03）；
- **领域 schema**：读到哪、卡住的概念、每份材料核心立场、材料间冲突、用户与材料的分歧、⭐权重（用户信号 > AI 回答）；
- 产品化的**"看看 AI 记了什么"查看面板**；
- **纠错交互**：人不直接编辑 memory，对话里说"你记错了"，AI 自己修正（memory 归 AI 所有，同 01 第三节的定稿）。

### 实现落点

纯文件 + 本地存储，零后端，TS 实现。通过 pi 工具暴露给 agent：`memory_read` / `memory_update` / `memory_search`。

- 小尾巴：embedding 来源待定（API 调用，或第一版直接 BM25 绕开）。

## 两家 memory 的本质差异（设计视角的记录）

表面看是功能清单不同，本质是两个产品面对的"遗忘问题"根本不是同一个问题。

### 各自要解决的问题

- **Claude Code 的问题是"每次开工都是新人"**。它是项目协作者，一场会话结束，下一场从零开始。但项目里大部分知识不需要记——代码、历史、文档都在仓库里随时能查。真正会丢的是**查不到的那一小撮**：用户是谁、他纠正过你什么、项目的隐性约束、外部资源在哪。所以它的 memory 为"少量、高价值、必须准确的事实"设计。
- **OpenClaw 的问题是"永不下班，但脑容量有限"**。它是常驻个人助理，活在消息流里，一天几十次琐碎交互，且上下文会被不断压缩——压缩就是失忆。它面对的是**大量、低密度、事后才知道哪条重要的信息流**，而且没有"仓库"兜底，记忆就是它的全部世界。

### 两种问题催生两种设计

- **Claude Code 在写入端做功，读取端偷懒**：写入时精挑细选——一事一文件、类型标签、重复就更新不新增、错了就删、明确的"不该记什么"清单（仓库能查到的不记）。写入端保证了 memory 小而准，读取端就简单到原始——一个索引文件常驻上下文，完事。没有向量、没有检索、没有衰减，因为策展过的小知识库不需要这些。**图书馆员模型：上架前分类编目，找书就快。**
- **OpenClaw 在读取端做功，写入端宽松**：写入接近日记——会话自动存档、每日笔记随手记，不苛求当场判断价值。代价是记忆体越滚越大、噪音多，于是复杂度全堆到读取侧：切块 embedding、向量 + BM25 混合检索、MMR 去冗、30 天半衰期让旧信息沉底，再加后台蒸馏循环定期把日记里的真金提炼进常青的 MEMORY.md。还有一个它独有的保命机制——**上下文快被压缩时触发静默回合，先把值得留的写下来再失忆**（"没有仓库兜底"的产品才需要这个）。**日记作者模型：先写下来再说，靠搜索和时间把重要的浮上来。**

一句话概括：**Claude Code 赌"写入时就能判断什么值得记"，OpenClaw 赌"只有事后才知道什么值得记"**。前者的赌注在项目协作场景成立（用户纠正你的那一刻，你就知道这很重要）；后者的赌注在生活助理场景成立（周二随口提的航班号，周五才变得重要）。

### 对本项目的启示：两个赌注各对一半

我们的场景恰好被切成两半，分别落在两家的假设里：

- **划线、批注、⭐、用户的明确观点**——价值在发生那一刻就确定了（你亲手划的线就是重要的），适用 Claude Code 的写入端策展：即时、类型化、去重；
- **陪读对话的 transcript**——一场几十轮的闲聊式问答，哪句将来有用当场说不准，适用 OpenClaw 的读取端管线：先完整存档，会话收尾静默蒸馏，长大了再上检索。

所以定稿方案"结构学 Claude Code、管线学 OpenClaw"不是折中——**我们的数据天然就有这两种性格，各配各的机制**。
