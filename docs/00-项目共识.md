## 一、产品愿景：主题阅读 AI Copilot

做一个 **AI 陪读软件**，不是读后摘要工具。区别在于：AI 参与的是"读的过程"，而不是读完之后的加工。

核心场景：

1. 用户围绕某个主题读书/读论文，遇到不懂的地方随时问 AI——选中文字、或截图/圈选发给 AI；
2. AI 解释之后，这些内容沉淀成结构化读书笔记；
3. 最终能基于笔记生成分享稿，甚至一键转 PPT。

产品形态设想：

- 左侧：书 / PDF 素材；
- 中间/侧边：AI 陪读对话；
- 另一侧：自动沉淀的笔记。

## 二、核心需求（所有选型决策的判断标准）

后面所有技术选型的取舍，都以这几条为准绳：

1. **主要交互是"划线" + "在划线内容下面挂批注"**。批注是打字，不是手写。
2. **格式必须以 PDF 为主**（将来会大量读论文），EPUB 读书为辅。
3. **使用设备：电脑 + iPad，两端都要能用 AI 交互**。iPad 上没有 AI 是不可接受的——这是**硬约束**，直接否掉了一条看起来很美的路线（见第四节 Zotero 插件路线）。
4. **Apple Pencil 主要当"选取笔"用**（划线、圈选），不需要 Notability 式的手写笔记能力。已**明确战略性放弃手写批注**——这条决定了不必为手写墨迹的顺滑度买单。
5. 一个有潜力的差异化交互点子：**"圈选问 AI"**——用 Pencil 圈一块区域，把该区域的文字/图像发给 AI 侧栏。技术上这是一个临时手势层：不存墨迹，因此没有排版锚定问题。

## 三、路线决策：二次开发，不从头写

理由：底座阅读器——排版、翻页、批注锚定这些东西——属于**"不加分但缺了不行"**的部分。自己写既耗时又做不过打磨多年的成品。所以白拿一个成熟的阅读批注引擎，把精力全部投入真正差异化的 AI 能力上。

配套原则：**不动底座原始功能，只加 AI 层，侵入点收敛到最少**。这样将来底座升级时不至于合并地狱。

## 四、选型演进过程（被否方案及原因）

### 1. 手写批注类开源软件（Saber / Xournal++ / Okular / Rnote）— 否

这类软件是**手写批注 / 无限画布导向**，与"AI 陪读"的方向错位。如果 fork 过来，它们的原有功能不是资产而是包袱。

### 2. Readest（Tauri 2 + Next.js + foliate-js）— 否

优点很突出：EPUB 体验好，iPad 有官方 App Store 版（也证明了 Tauri v2 打 iOS 包这条路走得通）。

但致命问题是 **PDF 是二等公民**：

- PDF 里连划线高亮都不支持（github.com/readest/readest/issues/2228，2025-10 提出至今未解决）；
- 手写批注功能为零，仅有一个未开发的 feature request（#3673）。

与"PDF 为主"的核心需求（第二节第 2 条）直接冲突。**退回为"纯 EPUB 场景备选"**，不作为主路线。

### 3. Zotero 官方 App + 插件路线 — 否

这是最可惜的一个方案。优点几乎全中：

- 批注模型完全命中需求：高亮 + 评论 + 批注面板 + 一键提取成笔记；
- PDF 能力是同类中最强的；
- iPad 官方 App 原生支持 Pencil；
- 有 zotero-gpt、PapersGPT 等 AI 插件先例，说明桌面端加 AI 是成熟路子。

但致命短板：**插件只能在桌面端跑，Zotero 的 iPad App 不支持插件**。这意味着 iPad 上永远没有 AI → 违反第二节第 3 条的硬约束。否。

### 4. 最终方案：嵌入 zotero/reader 引擎 + 自建应用壳 — 采纳

Zotero 路线否掉的是"官方 App + 插件"这个形态，但 Zotero 的批注引擎本身是对的。Zotero 官方把阅读批注组件独立开源为 [zotero/reader](https://github.com/zotero/reader)，可以直接嵌入自建应用——这样既拿到了 Zotero 级别的批注体验，又绕开了 iPad 插件限制。详见下一节。

## 五、最终技术方案

### 核心引擎：zotero/reader

- Zotero 官方独立开源的阅读批注组件；
- 纯 Web 技术（JS/TS，底层 pdf.js + epub.js）；
- 支持 PDF / EPUB / HTML；
- 已实现"划线 + 挂批注"交互；
- 构建产物含 `web` 变体（Zotero 网页版就是用它），生来就是可嵌入的。

### 应用壳（自建，交互完全自主）

- 书库/论文库管理；
- AI 陪读侧栏——**核心开发量所在**；
- 结构化笔记面板。

### 打包

一份 Web 代码：

- 桌面端用 Tauri / Electron 打包；
- iPad 用 Tauri v2 的 iOS 支持打包（先例：Readest 就是以此上架 App Store）。

由此 AI 侧栏在 iPad 上是**原生存在**的，Zotero 插件那套限制不复存在。

### 架构图

```
你自己的 App 壳（交互完全自主）
├─ 书库/论文库管理
├─ AI 陪读侧栏          ← 核心开发量
├─ 结构化笔记面板
└─ zotero/reader（嵌入） ← 阅读+划线+批注引擎
```

## 六、已知代价与风险（如实记录）

1. **壳的责任全部自担**：文件管理、批注数据存储、桌面/iPad 同步都要自己做。reader 只管渲染和产生批注事件。
2. **zotero/reader 嵌入 API 文档稀少**。需要读 Zotero 桌面版源码 + 跑自带 demo（`npm start` → `dev/reader.html`）来摸清接口。这是一次性学习成本。
3. **许可证 AGPL-3.0**：壳如果对外发布，也必须开源；自用无碍。
4. **手写墨迹手感有天花板**：Zotero iOS 官方 App 的 PDF/Pencil 墨迹用的是商业 SDK（PSPDFKit），不是开源的 reader。自建路线拿不到那种手写顺滑度，webview 级别的墨迹手感较差。但由于已战略性放弃手写批注（第二节第 4 条），核心交互——划线、打字批注、圈选问 AI——都不依赖高精度手写，影响可控。
